# word-addin

Stage-3 scaffold of a Word Task Pane add-in that converts Bijoy-encoded runs to
Unicode Bengali in place, using the `banglakit-wasm` crate as its engine.

> **Status: scaffold, not yet validated against a live Word host.** The Rust
> side (`banglakit-wasm`) is built and unit-tested. The Office.js side compiles
> with the dependencies in `package.json`, but actually running it requires
> a Windows / macOS machine with Word and `office-addin-debugging` installed,
> which is out of reach of the CI environment that produced this scaffold.

## Why this layout

- `manifest.xml` — the v1.1 task-pane manifest. Word reads this to decide what
  to load. Replace the placeholder `<Id>` GUID before publishing.
- `src/taskpane/taskpane.ts` — the Office.js entry point. The
  `convertRange` function is the meat: it iterates `getTextRanges` results,
  reads `font.name` + `text`, calls `convertRun` from the WASM bindings, and
  writes back via `insertText(..., Replace)`.
- `wasm-pkg/` — generated by `wasm-pack`. The webpack config copies the
  `.wasm` blob into the bundle and `taskpane.ts` imports the JS glue.
- `webpack.config.js` — sets `experiments.asyncWebAssembly: true` and pulls
  HTTPS certs from `office-addin-dev-certs` (Word will refuse to load a
  task pane from `http://`).

## First-time setup

```bash
cd examples/word-addin
npm install
npx office-addin-dev-certs install      # one-time, installs a local CA
npm run build:wasm                       # rust → wasm via wasm-pack
npm run dev-server                       # serves https://localhost:3000
```

In a second terminal:

```bash
npm start                                # sideloads manifest.xml into Word
```

`office-addin-debugging` will open Word with the task pane attached. Click
**Convert document** or select a region and click **Convert selection only**.

## What the run loop looks like

```ts
await Word.run(async (context) => {
  const range = context.document.body.getRange();
  const runs = range.getTextRanges([" ", "\t"], false);
  runs.load("items/text,items/font/name");
  await context.sync();                   // single round-trip for reads

  for (const r of runs.items) {
    const result = convertRun(r.text, r.font.name, "bijoy", "safe", "Kalpurush");
    if (result.changed) {
      r.insertText(result.text, Word.InsertLocation.replace);
      r.font.name = result.suggestedFont!;
    }
  }
  await context.sync();                   // single round-trip for writes
});
```

This is the same per-run, font-aware policy as the DOCX adapter — only the
adapter is different (Office.js instead of `quick-xml` + `zip`).

## Known gaps in this scaffold

- **No icons.** `manifest.xml` references `assets/icon-32.png` /
  `icon-80.png`; drop real PNGs into `assets/` before sideloading.
- **No tests.** Office.js can be mocked with `office-addin-mock` but the
  scaffold doesn't wire that up yet.
- **No CI.** `wasm-pack` + `npm run build` would slot into the existing
  GitHub Actions workflow when the team is ready.
- **`getTextRanges` is approximate.** It splits on whitespace, which is a
  decent proxy for `<w:r>` boundaries but not exact. A v0.2 of the add-in
  should walk `body.paragraphs` → `paragraph.getRange().getTextRanges([], …)`
  with a finer delimiter list, or use the still-preview
  `Word.ContentControl`-based APIs that expose runs directly.
- **Excel / PowerPoint.** Same WASM engine, different host APIs; left for a
  follow-up.

## Distribution

1. **Sideload (dev):** `npm start` — already wired up.
2. **AppSource:** generate a real GUID, host the bundle on a CDN with a
   stable HTTPS URL, submit through Microsoft Partner Center.
3. **Centralized deployment:** for newsroom / publisher orgs, upload the
   manifest in the Microsoft 365 admin center under
   *Settings → Integrated apps → Upload custom apps*. This is the most
   realistic distribution channel given the Bengali-publishing use case.
